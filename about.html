Dual System Architecture

ASPM and FLMA are parallel systems built on ServiceNow, each designed for distinct operational contexts. While they share underlying architectural principles, they maintain separate implementations to preserve domain-specific terminology, workflows, and requirements.

ASPM (Production Ready)

Enterprise Strategic Portfolio Management

Purpose: ServiceNow implementation and engine for civilian/enterprise strategic portfolio management

Context: Enterprise architecture, change management, and strategic planning for ServiceNow CTA/CMA consultants working with enterprise clients

Key Differentiator: Uses civilian/enterprise terminology and workflows optimized for commercial ServiceNow implementations

FLMA (In Development)

Field-Level Maintenance Architecture

Purpose: Digital twin for Army field-level maintenance operations

Context: Army maintenance operations with ServiceNow workflows as execution layer and GCSS-A as system of record

Key Differentiator: Uses Army-specific terminology and context, preserving military operational requirements without civilian terminology saturation

Why Separate Systems?

Maintaining separate systems ensures that:

  • Context Preservation: Each system maintains its domain-specific terminology and workflows without cross-contamination
  • Optimized Workflows: ASPM optimizes for enterprise ServiceNow implementations, while FLMA optimizes for Army maintenance operations
  • Clear Boundaries: Prevents civilian terminology from saturating military contexts and vice versa
  • Focused Development: Each system can evolve independently based on its specific domain requirements

FLMA System Overview

FLMA (Field-Level Maintenance Architecture) creates a true digital twin for Army field-level maintenance operations. The system architecture uses ServiceNow workflows as the maintenance execution layer while maintaining GCSS-A as the authoritative system of record.

FLMA Core Components

  • Digital Twin Architecture - Complete digital representation of Army field-level maintenance operations, processes, and workflows
  • ServiceNow Execution Layer - ServiceNow workflows provide the maintenance execution and orchestration layer
  • GCSS-A Integration - GCSS-A remains the system of record, with ServiceNow workflows integrating seamlessly
  • Army-Specific Context - Terminology, workflows, and processes tailored specifically for Army maintenance operations
  • Maintenance Workflow Engine - Specialized workflows for field-level maintenance operations

Development Status: FLMA is currently in active development. Core architecture and ServiceNow workflow integration are being developed to support Army field-level maintenance digital twin requirements.

ASPM System Architecture

The ASPM Agent System is built on a modular, extensible architecture with clear separation of concerns.

1. Agents

Base agent framework with capability-based architecture. 10 specialized agents for different domains.

2. Core Engine

Context management, data models, and services layer. Provides foundation for all agents.

3. Orchestrator

Agent routing, strategy building, and OKR management. Coordinates multi-agent workflows.

4. Personas

Communication style customization and tone application. 9 specialized personas for different contexts.

5. Safety Wrapper

ASPM-Guardian provides file write protection, module mutation checks, and safety registry.

6. AIE Engine

Agent Improvement Engine for evaluation, improvement suggestions, and prompt rewriting.

7. Governance Modules

PMO Value Ring, ADKAR Intake, SPM Pattern Library, and ITIL policies.

8. Knowledge Systems

Knowledge Vault, Knowledge Governance, Notebook Tools, and NotebookLM Integration.
